# 真实手机群控架构与执行方案

## 1. 目标与定位
- **核心诉求**：彻底绕过小红书 App 端的高级设备风控与虚拟机（模拟器）底层检测。
- **业务目标**：构建 10~50 并发级别的高稳定性矩阵运营系统，实现全链路自动化（养号、抓取、发布、互动）。

## 2. 硬件拓扑架构 (Hardware Topology)
- **中控核心**: 高性能 PC 或工作站（16核/32G以上推荐，用于运行大规模 `scrcpy` 投屏和 `ADB` 并发控制调度）。
- **连接中枢**: 工业级独立供电 USB 集线器（Hub）。必须购买带专用大功率电源的 10 口/20 口工业版（如西普莱等），防止主板供电不足导致设备频繁掉线。
- **终端节点**: 二手安卓真机。
  - **推荐机型**：Redmi K系列 / Note系列、Google Pixel 系列。
  - **前置要求**：必须支持解锁 Bootloader，方便获取 Root 权限或刷入精简类原生系统（如 LineageOS / 欧版 MIUI），去除系统广告及自带的杀后台机制。
- **网络隔离**: 
  - **方案 A（物理隔离）**：单机单卡（物联网大流量卡），配合自动化脚本定期切换飞行模式以更换动态 IP。
  - **方案 B（代理隔离）**：连接局域网 WiFi，在手机内部安装代理客户端（如 v2rayNG / Postern），配置高匿名的动态住宅 IP（Residential Proxy），实现全局透明代理。

## 3. 软件系统选型 (Software Stack)
- **底层通信**: `ADB (Android Debug Bridge)` 作为基石。
- **中控与监控**:
  - `Scrcpy`：低延迟投屏显示与人工介入控制。
  - `STF (Smartphone Test Farm)`：可选，用于大规模集群管理、监控设备心跳、电量与发热状态。
- **自动化执行引擎**: 
  - `Uiautomator2 (OpenATX)`：基于 Python 的远程 RPC 控制，精准抓取与操作 UI 组件树（DOM），小红书首选。
  - `Auto.js / Hamibot`：作为备选，利用无障碍服务（Accessibility），脚本直接在手机本地闭环运行。
- **行为反风控引擎 (Behavioral Anti-Detection)**:
  - 非匀速轨迹算法：放弃直线点击，采用贝塞尔曲线（Bezier）模拟人类物理滑动轨迹与触控压力。
  - 动态节律控制：注入真实随机时长的休眠与迟疑停顿时间。

## 4. 实施阶段计划 (Execution Phases)

### 阶段一：单节点原型验证 (POC)
- [ ] **硬件准备**：采购一台二手测试机，解锁 Root，刷入纯净系统。
- [ ] **基础连通性**：电脑端配置 ADB 环境，通过 USB 成功识别设备，运行 `scrcpy` 测试投屏帧率与操控延迟。
- [ ] **引擎植入**：Python 环境安装 `uiautomator2`，通过 `python -m uiautomator2 init` 为手机植入守护进程（ATX 客户端）。
- [ ] **核心链路跑通**：编写第一个测试脚本完成闭环（例如：点亮屏幕 -> 打开小红书 -> 搜索关键词 -> 点击首篇笔记 -> 滑动停留 10 秒 -> 退出）。

### 阶段二：反风控与异常处理开发
- [ ] **物理轨迹封装**：开发高级触控组件，基于三次贝塞尔曲线生成离散坐标点集，配合 `u2.touch` 模拟非线性拖拽与滑动。
- [ ] **全局异常拦截池**：针对自动化过程中随时弹出的“青少年模式”、“App更新”、“隐私协议”、“滑块验证码”编写全局守护线程（Watcher）进行捕获和自动关掉。
- [ ] **环境隔离验证**：配置代理网络后，抓包（如 Charles 或 HTTP Toolkit）严格核查小红书 App 流量是否 100% 走代理出口，绝不能发生 DNS 泄露或局域网真实 IP 暴露。

### 阶段三：群控中台落地与规模化 (Scale Up)
- [ ] **规模扩容**：接入工业级 USB Hub，首批扩展至 5-10 台设备并行测试。
- [ ] **设备池化管理**：利用 Python 动态读取 `adb devices -l` 解析设备串号（SN），建立本地 SQLite 或 Redis 映射表（`SN -> 手机编号 -> 绑定的账号 -> 当前状态`）。
- [ ] **任务调度中枢**：基于 Celery 或轻量级多线程/多进程，将“养号大盘”、“指定发文”等需求拆解为原子任务，向空闲设备派发队列指令。
- [ ] **健壮性优化**：处理高并发下的 `ADB Server` 崩溃问题；实现设备掉线的自动重连报警机制。

## 5. 风险防范与安全基线
1. **网络铁律**：严禁所有设备在无代理保护的情况下连接同一办公网络直接向小红书发起请求，防止全军覆没的连坐封禁。
2. **账号-设备绑定**：坚持一号一机原则。频繁切换账号是大忌。配合 `LSPosed` 和 `Device Emulator` 等框架定期抹除特定缓存或调整少量环境参数（非必要不刷机）。
3. **物理安全（消防）**：长期插电高频运行极易导致手机电池鼓包，必须拆除手机后盖/电池（改直供电）或在机柜配备强力工业散热风扇，严防火灾隐患。
