# 小红书 (XHS) 防风控交互式截流自动化架构设计

## 1. 架构概述 (Overview)
传统的 Web 自动化脚本（如 Selenium/Puppeteer 控制脚本）高度依赖 DOM 操作和固定模式的时间休眠，极易被各大平台（如小红书）的前端探针和后端行为模型识别为机器人。

为了实现长效、安全的“地陪截流”或类似获客业务，本系统摒弃了“单体线性脚本”，重构为 **“AI 大脑 (Antigravity) + 仿生物理驱动器 (Browser Harness)”** 的读写分离双层架构。

## 2. 核心架构设计 (Core Architecture)

系统严格按照以下四个阶段 (Phases) 闭环运作：

### Phase A: 扫描与养号 (Scan & Farming)
*   **动作**：在信息流中扫描可视区域内的帖子及其坐标。
*   **防风控策略（业务级）**：强制引入**养号逻辑 (Zombie Surfing)**。在连续的业务截流操作间隙，AI 会随机点击无关帖子（如搞笑、日常分享），强制停留 10-15 秒进行“无效浏览”后再退出。此举旨在稀释账号的业务操作纯度，在后端风控模型中建立“正常活跃用户”的画像。

### Phase B: 感知与深度抓取 (Perception & Extraction)
*   **动作**：根据坐标进入目标帖子，抓取正文和评论树。
*   **防风控策略（交互级）**：引入了**视口突破机制**。由于长图文可能将评论区挤出屏幕下方，脚本在提取前会通过注入 JS 自动寻找特定的滚动容器（如 `.note-scroller`），执行平滑向下滚动，确保渲染出完整的评论 DOM。

### Phase C: 认知、意图风控与话术生成 (Cognitive Fuse & Generation)
*   **动作**：AI 大脑阅读提取的数据，决定是否回复，并动态生成话术。
*   **防风控策略（内容与公关级）**：
    *   **动态话术**：废弃死板的预设话术库。每次评论均由 LLM 结合上下文（如“楼中楼”逻辑、时间、地点要求）动态生成，避免触发后端文本哈希去重风控。
    *   **情绪熔断机制 (Sentiment Fuse)**：在生成话术前，强制进行情绪判定。一旦识别出帖子属于“避雷贴”、“吐槽求助贴”或带有强烈的负面情绪，系统立刻触发熔断，拒绝生成话术，转而执行闭包退出操作，彻底避免引发公关危机。

### Phase D: 物理动作与显式状态闭环 (Physical Act & Verification)
*   **动作**：进行物理点击和键盘打字。
*   **防风控策略（底层协议级）**：
    *   **贝塞尔曲线轨迹 (Bézier Curve Physics)**：彻底重写底层 `click_at_xy`。禁止鼠标光标瞬移。在每一次点击前，利用二次贝塞尔曲线算法计算中间坐标路径，并通过 `dispatchMouseEvent(mouseMoved)` 伴随 `5-15ms` 的微休眠，完美模拟人类手腕移动的加减速与颤动。
    *   **显式状态校验与异常恢复**：废除所有盲目的 `time.sleep()` 强等逻辑。在加载帖子、聚焦输入框时，采用高频 JS 轮询验证 DOM 状态。
    *   **结果三态闭环**：点击“发送”后，进行最终校验：
        1.  `SUCCESS`：提取 DOM 确认发送的文本已挂载。
        2.  `CAPTCHA`：页面出现“安全验证/滑块”等字眼，系统立即中止抛出警报。
        3.  `DRAFT_STILL_EXISTS`：输入框未清空，记录网络或拦截错误。

## 3. 防风控铁律规范总结 (The Red Lines)

本系统的安全性基于对以下开发红线的绝对遵守：
1.  **禁止 DOM 触发交互**：全流程代码中不得出现 `element.click()` 或 `input.value = ""`。所有导致状态改变的交互必须通过坐标 `click_at_xy` 和拟人打字 `human_type` 完成。
2.  **感知与执行隔离**：感知层（通过 `js()` 提取 DOM 信息）只能用于“看”。“看”到的结果（如按钮坐标）必须传递给物理层进行“动”。
3.  **无状态环境规避**：必须挂载用户真实使用的 Chrome 实例（携带日常 Cookie 和 LocalStorage），严禁使用纯净无头的 Puppeteer 实例启动自动化。
